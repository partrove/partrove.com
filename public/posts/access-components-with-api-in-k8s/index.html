Cluster API#

获取证书#

使用 kubeadm 搭建的集群，默认会给 kubectl 生成一份 admin.conf 的 kubeconfig，位置在 /etc/kubernetes/admin.conf ，其中就有可使用的证书信息

mkdir /tmp/certs
grep client-certificate-data /etc/kubernetes/admin.conf | awk '{print $2}' | base64 -d > /tmp/certs/cert.crt
grep client-key-data /etc/kubernetes/admin.conf | awk '{print $2}' | base64 -d > /tmp/certs/cert.key

使用 curl 访问集群 API#

curl 命令中，带上前面生成的证书，即可访问集群 API

curl --cacert /etc/kubernetes/pki/ca.crt --cert /tmp/certs/cert.crt --key /tmp/certs/cert.key https://192.168.132.10:6443/healthz

通过集群 API 删除 Terminating 的 namespace#

集群内删除 namespace 时，删除过程中可能会一直处于 Terminating 状态

[root@master]# kubectl get ns
NAME              STATUS        AGE
apps              Terminating   3h26m
default           Active        37h
kube-node-lease   Active        37h
kube-public       Active        37h
kube-system       Active        37h
kyanite           Active        3h6m
monitoring        Active        3h58m

此时除了 delete 时使用 --force --grace-period 0 参数，还可以 edit 资源对象，然后删除 finalizer 再保存。 如果这两个方法都无法删除 Terminating 的 namespace，最后还可以通过调用集群 API 来请求删除。
首先导出需要删除的 namespace 资源对象，导出格式为 json。

kubectl get ns apps -ojson > /tmp/apps-ns.json

然后编辑导出的 json 文件，将 finalizer 列表中的 Kubernetes 行删除掉

vim /tmp/apps-ns.json
 #   "spec": {
 #       "finalizers": [
 #           "Kubernetes",    # 删掉这一行
 #       ]
 #   },

保存编辑后的 json 文件，然后调用集群的 /api/v1/namespaces/apps/finalize 接口

curl --cacert /etc/kubernetes/pki/ca.crt --cert /tmp/certs/cert.crt --key /tmp/certs/cert.key -H "Content-Type: application/json" -X PUT --data-binary @apps-ns.json https://192.168.132.10:6443/api/v1/namespaces/apps/finalize

其中 /apps 对应要删除的 namespace 名称。 调用成功后，再查看 ns 列表，可以看到已成功删掉。

[root@master tmp]# kubectl get ns
NAME              STATUS   AGE
default           Active   37h
kube-node-lease   Active   37h
kube-public       Active   37h
kube-system       Active   37h
kyanite           Active   3h13m
monitoring        Active   4h5m

https://www.redhat.com/sysadmin/troubleshooting-terminating-namespaces

Kubelet#

使用 kubeadm 搭建的集群，kubelet 默认会监听了 127.0.0.1:10248 和 *:10250 两个端口，其中 10248 端口上提供了 /healthz 健康检查接口。而在 10250 端口上，则提供了一系列 RESTful API，例如 /metrics 和 /pods 等。

https://www.deepnetwork.com/blog/2020/01/13/kubelet-api.html

kubelet 接口默认启用了认证，访问需要带上证书。并且默认情况下服务使用了自签名证书，因此可以使用如下命令访问

curl -k --cacert /etc/kubernetes/pki/ca.crt --cert /etc/kubernetes/pki/apiserver-kubelet-client.crt --key /etc/kubernetes/pki/apiserver-kubelet-client.key https://192.168.132.10:10250/metrics

如果要不带 -k 参数访问，则需要首先配置 kubelet 使用集群 ca 签名证书。可参考官方文档
